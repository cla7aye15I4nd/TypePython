# TypePython Test Suite

## Overview

This test suite uses an **automated test discovery system** - no manual test registration required!

## How It Works

### Adding New Tests

1. **Valid Tests**: Create a `.py` file in `tests/fixtures/valid/`
   - The test will be automatically discovered and run
   - Expected output (`.txt` file) is auto-generated by running Python3 during test execution
   - No manual test registration needed!
   - `.txt` files are gitignored - they're generated on-the-fly

2. **Invalid Tests**: Create a `.py` file in `tests/fixtures/invalid/`
   - Must be code that fails in both our compiler AND Python3
   - Automatically discovered and verified
   - No test registration needed!

3. **Ignored Tests**: Put files in `tests/fixtures/valid/ignored/`
   - These tests are known to fail (unimplemented features, bugs, etc.)
   - Automatically generated with `#[ignore]` tag
   - Not run in normal test execution
   - Can be run with `cargo test --test integration_tests -- --ignored`

## Running Tests

```bash
# Run all tests (recommended: single-threaded to avoid file conflicts)
cargo test --test integration_tests -- --test-threads=1

# Run all tests (parallel - faster but may have file conflicts)
cargo test --test integration_tests

# Run only valid tests
cargo test --test integration_tests test_basic
cargo test --test integration_tests test_advanced

# Run only invalid tests
cargo test --test integration_tests test_invalid

# Run ignored tests
cargo test --test integration_tests -- --ignored

# Run specific test
cargo test --test integration_tests test_basic_hello_world

# Check code coverage
cargo llvm-cov --summary-only --test integration_tests -- --test-threads=1
```

## Test Organization

```
tests/
├── fixtures/
│   ├── valid/          # Tests that should compile and run correctly
│   │   ├── basic/      # Basic language features
│   │   ├── advanced/   # Advanced features
│   │   ├── builtins/   # Builtin functions
│   │   ├── dict/       # Dictionary operations
│   │   ├── list/       # List operations
│   │   ├── set/        # Set operations
│   │   ├── modules/    # Module import tests
│   │   ├── strings/    # String/bytes operations
│   │   └── ignored/    # Tests with known issues (auto-tagged with #[ignore])
│   └── invalid/        # Tests that should fail compilation
├── integration/
│   ├── auto_tests.rs   # Auto-generated test module (includes build.rs output)
│   ├── cli_tests.rs    # CLI interface tests
│   ├── mod.rs          # Module declarations
│   └── test_runner.rs  # Test execution logic
└── README.md           # This file

build.rs                # Generates test code at build time
```

## Module Tests

Module test files have special handling:
- Main test file: Must be named `main.py`, `test.py`, or start with `test_`
- Helper modules: Any other `.py` files (won't be run as standalone tests)

Example:
```
modules/simple_import/
├── main.py        # Main test file (runs)
└── math.py        # Helper module (not run as test)
```

## Benefits of Automated Testing

1. **Zero Manual Work**: Just add a `.py` file - that's it!
   - No test registration in Rust code
   - No `.txt` files to maintain (auto-generated)
   - No `.out` files to track (auto-cleaned)

2. **Build-Time Code Generation**:
   - `build.rs` scans for test files during compilation
   - Generates individual Rust test functions (300+ tests)
   - Each `.py` file becomes its own test function

3. **Auto-Generated Expected Outputs**:
   - First test run: Python3 generates `.txt` file
   - Subsequent runs: Compares against `.txt` file
   - `.txt` files are gitignored (not committed)

4. **Automatic Ignored Test Handling**:
   - Files in `/ignored/` folder get `#[ignore]` tag
   - No manual test attribute management needed

5. **Fast Iteration**:
   - Add test → Rebuild → Run
   - No Rust code editing required
   - Self-documenting test organization
